# wiki-ingest

지정된 **소스 폴더 하나**(프로젝트 회의록 폴더 또는 `15. Training\15.02. Technical` 기술 학습자료 폴더)를 입력으로 받아 아래 파이프라인을 실행한다. 한 번에 폴더 하나만. 수동 호출 전용.

위키 루트: `C:\Users\Z006K14G\Desktop\Yejun\90. Wiki`

> **위키의 목적 = 학습/연결성** (Core Context §2-0). 이 위키는 단순 색인이 아니라 **공부용 백과사전**이다. 각 노트는 (1) **원본을 안 열어도 이해되는 자기완결 본문**, (2) 인접 개념으로 항해 가능한 **풍부한 `## 관련` 연결**을 갖춰야 한다. 한 줄 정의 stub은 목적 미달 — 소스의 깊은 내용을 본문으로 종합하라.

## 절차
0. **Core Context 먼저 읽기**: 위키 루트의 `Core Context.md`가 있으면 읽고, 거기 정의된 **재활용 축**(이 위키가 어디에 쓰일지: 제안 / 견적 / 기술자문 / 고객대응 / 교훈 등)과 원칙에 맞춰 "무엇을 추출할 가치가 있나"를 판단한다. 없으면 이 단계는 생략(처음 만들 땐 `references/Core Context.template.md`를 볼트 루트에 복사해 채우도록 안내).
1. 소스 노트를 읽는다: MOC(`0_*.md`)·이벤트 노트(`YYMMDD_*.md`)·기술 학습 노트. **본문만 보고 끝내지 말 것** — ① 노트에 **첨부된 HTML 파일**(예준님은 메일·자료를 주로 HTML로 정리함)을 반드시 읽고, ② 노트 안 `file:///` 링크로 **폴더에 연결된 자료**(U:\ 첨부·이미지·문서)도 따라가 읽는다. 이게 지식 본문의 1차 출처.
2. 엔티티 후보(고객사·사람·설비/제품)와 개념 후보(기술 용어·영업/AI 개념)를 추출한다.
3. 노트를 만들기 전에 **추출 후보를 표로 사용자에게 먼저 보고하고 확인받는다**(생성 게이트).
4. `90. Wiki\Entities`와 `90. Wiki\Concepts`에서 **기존 노트 중복 체크 + 연결될 개념 검색**.
   - 있으면 갱신(얇은 stub이면 백과사전형으로 **재작성** + 새 출처 백링크 추가), 없으면 새로 생성.
   - 새 노트가 연결될 만한 기존 노트를 먼저 검색해 `## 관련` 백링크를 적극적으로 건다.
5. 원자 노트 작성 규칙(한 노트 = 한 개념, **백과사전형**):
   - 파일명: 개념명 그대로(예: `P-Mix.md`, `POSCO P2C.md`).
   - frontmatter `description:` 한 줄(note-description 컨벤션 따름).
   - 태그: 엔티티는 `entity/...`, 개념은 `topic/...`.
   - frontmatter `confidence:` (`high`/`medium`/`low`) — 추출 근거의 신뢰도.
   - frontmatter `explored: false` — LLM이 만든 직후 기본값. 사람이 확인했거나 별도 검증을 거치면 `true`로.
   - frontmatter `created:` — 노트를 처음 만든 날짜(`currentDate`, YYYY-MM-DD). **새로 만들 때만 기록하고, 기존 노트 갱신 시에는 절대 덮어쓰지 않는다**(없으면 그때 한 번 채워도 됨).
   - frontmatter `reusableFor:` 한 줄(선택) — Core Context 재활용 축 중 어디에 쓰일지(예: `학습/기술자문`, `제안`, `견적`).
   - `confidence: high`로 둘 땐 본문에 `> [!note] Bias Check` 한 줄(반대해석 또는 데이터 공백) 추가.
   - **본문 = 자기완결 백과사전 항목**: 정의 → 원리·구조·세대/종류 → 핵심 트레이드오프·수치 → 실무/POSCO 포인트. 소스(첨부 HTML·`file:///` 자료·Training 노트)의 깊은 내용을 종합하되 원본 안 열어도 이해되게. 한 줄 stub 금지.
   - 끝에 `## 출처`(소스 노트로 `[[백링크]]`, 원본 링크 보존) → `## 관련`(연관 노트 `[[링크]]`, 없으면 만들 예정 개념도 forward link로).
6. 이벤트 노트 본문에 해당 원자 노트로 `[[wikilink]]`를 추가(원문·출처 링크는 보존).
7. 프로젝트 MOC의 요약 섹션 갱신: `2.1 Sales Activity`(시간순 요약), `5. Learning from failure`(교훈), 관련 원자 노트 `[[링크]]`.
8. `90. Wiki\0_Wiki MOC.md`의 Entities/Concepts 인덱스 갱신.
9. **한 일 보고**: 생성/갱신한 원자 노트, 추가한 백링크, 갱신한 MOC 섹션을 목록으로.
10. **실행 로그 남기기**: `90. Wiki\_ingest-log.md`에 이번 실행 1행을 append(맨 아래).
    - 파일이 없으면 헤더와 함께 새로 만든다:
      ```markdown
      # Ingest Log

      | 날짜 | 프로젝트 | 생성 | 갱신 | 백링크 | MOC 갱신 | 생성 노트 | 갱신 노트 | 비고 |
      |------|----------|------|------|--------|----------|-----------|-----------|------|
      ```
    - 한 행 형식: `| 2026-06-19 | <프로젝트 폴더명> | <생성 수> | <갱신 수> | <백링크 수> | <갱신한 MOC 섹션 수> | <생성한 노트 [[링크]] 쉼표로 나열> | <갱신한 노트 [[링크]] 쉼표로 나열> | <확인필요/모순 등 특이사항, 없으면 -> |`
    - **생성/갱신 노트** 열에는 실제 노트 이름을 `[[링크]]`로 모두 적는다(없으면 `-`). "어제 무슨 노트 만들었나"를 로그만 보고 답할 수 있어야 한다.
    - **append만**: 기존 행은 절대 수정/삭제하지 않는다.
    - 날짜는 `currentDate`(YYYY-MM-DD). lint 단독 실행 시에는 비고에 `lint`로 표기하고 생성/갱신/백링크는 `0`, 노트 열은 `-`.

## Lint (건강도 체크 — 단독 호출 또는 ingest 후 마지막 단계)
트리거: "위키 lint", "위키 건강도 체크", "위키 정리 점검". **검사만 하고 보고**(자동 수정 금지, 생성 게이트와 동일하게 사용자 확인 후 수정).
- **Orphan**: `90. Wiki\Entities`·`Concepts`에서 아무 노트·MOC에도 링크 안 된 원자 노트.
- **깨진 링크**: 본문 `[[wikilink]]` 중 실제 파일이 없는 것.
- **모순**: 같은 엔티티/개념에 상충하는 서술 → 보고하고 해당 노트에 `> [!warning] Contradiction` 제안.
- **MOC 동기화**: `0_Wiki MOC.md`의 Entities/Concepts 인덱스 ↔ 실제 노트 목록 불일치.
- **품질 backlog**: `explored: false`로 오래 남은 노트, `confidence: high`인데 Bias Check 없는 노트, `description` 누락.
- 결과는 표로 보고 → 사용자가 고를 항목만 수정.

## 질의(Ask — 검색·요약, 읽기 전용)
트리거: "위키에서 ~ 찾아줘", "~가 뭐였지", "~ 문제가 뭐였지", "어제/최근 만든 노트 요약". **읽기만 하고 노트는 수정하지 않는다**(`_ingest-log.md`에도 안 남김).
1. **검색**: `90. Wiki\Entities`·`Concepts`와 프로젝트 폴더에서 검색어(엔티티명·개념명·동의어)를 찾는다.
2. **출처 따라가기**: 매칭된 원자 노트의 `## 출처` 백링크를 따라 원본 이벤트 노트(`YYMMDD_*.md`)에서 실제 내용을 읽는다.
3. **시간 필터**: "어제/최근/이번 주" 같은 표현이 있으면 노트 frontmatter `created:`(없으면 파일 수정일)와 이벤트 노트 날짜(`YYMMDD`)로 거른다. "어제 만든 노트"는 `_ingest-log.md`의 해당 날짜 행에서 생성/갱신 노트 목록을 바로 참고해도 된다.
4. **종합·요약**: 개조식(~함)으로 답하고, **근거가 된 노트·이벤트 노트를 `[[링크]]`로 모두 표기**한다.
5. **신뢰도 반영**: 근거 노트의 `confidence`가 `low`거나 출처가 빈약하면 "확인 필요"로 표시하고 단정하지 않는다. 모순되는 서술이 보이면 양쪽을 같이 제시한다.

## 제약(안전장치)
- 회의록의 `# Tiro`·`# Yejun's memo` 섹션은 **건드리지 않는다**. Claude 메모는 `# Claude Code` 섹션에만.
- `0_` 노트는 자동 태깅 대상에서 제외.
- 한국어 회의록 요약/번역은 개조식(~함 종결).
- 원본 .msg/U:\ 링크는 보존, 끊지 않는다.
- 확신 없는 추출은 만들지 말고 보고에 "확인 필요"로 남긴다.
- `_ingest-log.md`는 **append 전용** — 과거 실행 기록은 절대 고치거나 지우지 않는다.
