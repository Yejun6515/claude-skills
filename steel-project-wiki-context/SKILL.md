# steel-project-wiki-context

철강 업계 안건(고객사·라인·설비 프로젝트)이 대화에 등장하면, **답하거나 작업을 시작하기 전에 먼저 옵시디언 업무 위키에서 그 안건의 현재 상황을 확인**한다. 사용자가 "위키 봐"라고 말하지 않아도, 철강 안건이 화제가 된 것 자체가 트리거다.

- 볼트 루트: **PC마다 다름** — `%USERPROFILE%\.claude\skills\_config\local-paths.md`의 `vault_root:` 값(없으면 사용자에게 물어 저장 — `_config\README.md`)
- 위키 루트: `90. Wiki` · 링크 사전(인덱스): `90. Wiki\0_Wiki MOC.md`
- 프로젝트 폴더: `01. Projects\` (예: `01.02 POSCO\POSCO_K3C_Revamp\`)

> **왜 먼저 확인하나**: 위키 엔티티 노트와 프로젝트 MOC에는 메모리·일반지식에 없는 **현재 진행상황·마스터 스케줄·다음 할 일**이 들어 있다. 기억이나 추측으로 답하면 철 지난·틀린 맥락으로 일하게 된다. 위키가 single source of truth.

## 절차

1. **트리거 인지** — 메시지에 철강 고객사/라인코드/철강 설비 프로젝트가 등장하면 발동. 사용자가 위키를 언급하지 않아도, "이거 알아?"가 아니라 "이거 해줘"여도 **먼저 위키부터 확인**한다.
2. **인덱스 매칭** — `90. Wiki\0_Wiki MOC.md`를 읽어 언급된 엔티티·라인코드·개념이 위키에 있는지 찾는다. 약어·동의어·표기변형 고려: **K3CX=POSCO K3C, HTC=Hitachi, P2H=포항2열연, MISCO/MIRECO=Nireco, MK valve=Moog** 등.
3. **상세 읽기** — 매칭된 노트를 실제로 읽는다(제목만 보고 답하지 않는다):
   - 위키 엔티티/개념 노트: `90. Wiki\Entities\<이름>.md`, `90. Wiki\Concepts\<이름>.md` — 정의·스코프·마스터 스케줄·관련 링크.
   - 프로젝트 MOC: 해당 `01. Projects\...\0_*.md` — Overview·Documents·To-dos·stakeholders.
   - **최근 이벤트 노트**: 같은 프로젝트 폴더에서 날짜가 가장 최신인 노트 1~3개 — 가장 최근 진행상황·미결 액션.
4. **상황 요약 후 진행** — 확인한 내용을 **현재 상황 / 마스터 스케줄(임박 일정) / 다음 할 일**로 짧게 정리한 뒤, 사용자가 원래 요청한 작업(견적·기술자문·고객대응 등)을 그 맥락 위에서 이어서 한다. 단순 질문이면 요약만으로 답이 된다.
5. **위키에 없을 때** — 인덱스에 없으면 프로젝트 폴더를 grep(파일명·`description:` frontmatter)으로 한 번 더 찾는다. 그래도 없으면 "위키/볼트에 해당 안건 노트가 없다"고 알리고, 일반지식 기반임을 밝힌 뒤 진행한다. 새 위키 노트를 임의로 만들지 않는다(생성은 `wiki-ingest` 소관).

## 검색 팁

- **description-first**: 어느 노트가 맞는지 애매하면 `Grep`으로 frontmatter `description:`를 훑어 후보를 좁힌 뒤 해당 노트를 연다.
- 라인 클러스터가 인접 라인을 참조할 수 있다(예 K3CX ↔ P1CX/P2C). MOC의 `## 관련` 링크를 따라가면 빠르다.
- 영문/국문/일문 표기가 섞인다 — `-i`(대소문자 무시)로 검색.

## 제약

- **위키/프로젝트 노트는 읽기 전용** — 이 스킬은 확인만 한다. 노트 작성·정리는 `note-digest`/`note-description`, 위키 갱신은 `wiki-ingest` 소관.
- **인물 정보는 위키 노트로 없음** — 담당자는 프로젝트 MOC 본문(stakeholders) 텍스트로 확인.
- 확인은 빠르게 — 인덱스+엔티티 노트+최근 노트 1~3개면 충분. 폴더 전체를 다 읽지 않는다.

## 발동 / 비발동

| 상황 | 발동? |
|---|---|
| "K3CX 견적 도와줘" / "포스코 P3PL 어떻게 되고 있어?" | ✅ 먼저 위키 확인 |
| "현대제철 신규 건 자료 정리해줘" (위키에 있든 없든) | ✅ 위키·폴더 먼저 조회 |
| 철강사·라인코드가 메시지에 그냥 나옴 | ✅ 확인 후 진행 |
| 철강과 무관한 작업(일본어 단어 TTS, 드라마 엑셀 등) | ❌ 발동 안 함 |

## Red flags — 이러면 멈추고 위키부터 확인

- "사용자가 위키 보라고 안 했으니 그냥 아는 대로 답하자" → **틀림**. 철강 안건 언급 자체가 트리거.
- "메모리/직전 대화에 맥락 있으니 위키는 건너뛰자" → 위키에 **현재 진행상황**이 있다. 임박 일정·미결 액션은 거기서만 최신.
- "간단한 질문이라 인덱스만 봐도…" → 인덱스 매칭 후 **해당 엔티티 노트는 실제로 읽는다**.
- "노트 제목만 봐도 알겠다" → 제목 추측 금지. 본문의 스케줄·다음 할 일을 읽는다.

## 자기점검 (rationalization 차단)

| 변명 | 실제 |
|---|---|
| "명시적으로 위키 확인하라고 안 했다" | 철강 고객사/프로젝트 언급 = 트리거. 지시 불필요. |
| "내가 이미 이 프로젝트 안다" | 안다고 믿는 건 과거 스냅샷. 마스터 스케줄·미결은 위키에서 재확인. |
| "작업만 빨리 해주면 된다" | 틀린 맥락 위 작업은 다시 해야 한다. 30초 확인이 비용 절약. |
| "위키에 없을 것 같다" | 추측 말고 인덱스·폴더 grep으로 실제 확인. 없으면 그때 밝히고 진행. |
